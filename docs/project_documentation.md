# Часть 1: План разработки по методологии SCRUM

## 1. Подготовительный этап (Pre-Game)

### 1.1 Формирование команды
- **Product Owner** — владелец продукта, определяет требования и приоритеты
- **Scrum Master** — фасилитатор процесса, устраняет препятствия
- **Development Team** — команда разработки (3-9 человек)

### 1.2 Создание Product Backlog
- Собрать все требования к системе
- Описать функционал в виде User Stories (формат: "Как [роль], я хочу [функция], чтобы [ценность]")
- Приоритизировать истории (MoSCoW или числовой приоритет)
- Оценить сложность каждой истории (Story Points)

### 1.3 Определение Definition of Done (DoD)
- Критерии готовности задачи (код написан, тесты пройдены, документация обновлена и т.д.)

---

## 2. Планирование спринта (Sprint Planning)

### 2.1 Выбор длительности спринта
- Обычно 1-4 недели (рекомендуется 2 недели)

### 2.2 Sprint Planning Meeting
- Выбрать истории из Product Backlog для текущего спринта
- Декомпозировать истории на задачи (tasks)
- Сформировать Sprint Backlog
- Определить цель спринта (Sprint Goal)

---

## 3. Выполнение спринта (Sprint Execution)

### 3.1 Daily Scrum (ежедневные стендапы)
- Длительность: 15 минут
- Каждый участник отвечает на 3 вопроса:
  1. Что сделал вчера?
  2. Что планирую сделать сегодня?
  3. Есть ли препятствия?

### 3.2 Работа над задачами
- Использовать Scrum-доску (To Do → In Progress → Done)
- Обновлять статус задач ежедневно
- Вести Burndown Chart для отслеживания прогресса

---

## 4. Обзор спринта (Sprint Review)

### 4.1 Демонстрация результатов
- Показать работающий функционал заказчику/стейкхолдерам
- Собрать обратную связь
- Обновить Product Backlog при необходимости

---

## 5. Ретроспектива спринта (Sprint Retrospective)

### 5.1 Анализ процесса
- Что прошло хорошо?
- Что можно улучшить?
- Какие действия предпринять для улучшения?

---

## 6. Артефакты SCRUM

| Артефакт | Описание |
|----------|----------|
| Product Backlog | Приоритизированный список всех требований |
| Sprint Backlog | Задачи, выбранные для текущего спринта |
| Increment | Готовый к релизу прирост продукта |
| Burndown Chart | График сгорания задач |

---

## 7. Рекомендуемые инструменты

- **Trello / Jira / YouTrack** — для ведения backlog и доски задач
- **Miro / Figma** — для прототипирования и визуализации
- **Git** — для контроля версий
- **Confluence / Notion** — для документации

# Часть 3: Definition of Done (DoD)

## Критерии готовности задачи

Задача считается выполненной (Done), когда выполнены ВСЕ следующие критерии:

---

## Для Backend (Rust/Rocket)

### Код
- [ ] Код написан и компилируется без ошибок (`cargo build`)
- [ ] Код соответствует стилю Rust (проверка через `cargo fmt`)
- [ ] Код проходит статический анализ (`cargo clippy` без warnings)
- [ ] Реализована обработка ошибок (нет unwrap() в продакшн коде)
- [ ] Код покрыт unit-тестами (минимум 70% покрытия для бизнес-логики)

### API
- [ ] Endpoint реализован согласно спецификации
- [ ] Реализована валидация входных данных
- [ ] Возвращаются корректные HTTP-статусы
- [ ] Реализована авторизация для защищённых эндпоинтов

### База данных
- [ ] Написаны и применены миграции
- [ ] Миграции работают в обе стороны (up/down)

---

## Для Frontend (React)

### Код
- [ ] Компонент реализован и работает без ошибок в консоли
- [ ] Код проходит ESLint проверку
- [ ] TypeScript типы определены (нет any)

### UI/UX
- [ ] Интерфейс соответствует макету от дизайнера
- [ ] Реализована адаптивность (desktop, tablet, mobile)
- [ ] Реализованы состояния загрузки (loading states)
- [ ] Реализована обработка ошибок (error states)

### Интеграция
- [ ] Компонент корректно взаимодействует с API
- [ ] Данные корректно отображаются и обновляются

---

## Для дизайна

- [ ] Макет создан в Figma
- [ ] Макет согласован с Product Owner
- [ ] Определены все состояния элементов (hover, active, disabled, error)
- [ ] Макет передан frontend разработчику

---

## Общие критерии

### Документация
- [ ] API задокументирован (Swagger/OpenAPI)
- [ ] Обновлён README при необходимости

### Code Review
- [ ] Код прошёл review от другого члена команды
- [ ] Все замечания исправлены

### Тестирование
- [ ] Функционал протестирован вручную
- [ ] Тесты проходят успешно
- [ ] Нет регрессий в существующем функционале

### Git
- [ ] Код закоммичен в feature-ветку
- [ ] Создан Pull Request
- [ ] PR смёржен в develop ветку

---

## Definition of Done для спринта

Спринт считается завершённым, когда:

- [ ] Все взятые в спринт User Stories соответствуют DoD
- [ ] Проведён Sprint Review с демонстрацией функционала
- [ ] Проведена Sprint Retrospective
- [ ] Product Backlog обновлён по результатам обратной связи
- [ ] Increment готов к потенциальному релизу

---

## Definition of Done для релиза

Релиз готов, когда:

- [ ] Все запланированные фичи реализованы и соответствуют DoD
- [ ] Проведено интеграционное тестирование
- [ ] Проведено приёмочное тестирование с Product Owner
- [ ] Документация актуальна
- [ ] Приложение развёрнуто на staging окружении
- [ ] Проведено smoke-тестирование на staging
